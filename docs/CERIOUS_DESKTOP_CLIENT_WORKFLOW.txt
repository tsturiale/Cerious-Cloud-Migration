Cerious Systems Desktop Client Workflow
Stable local baseline: stable-local-20260616-091747

Purpose
Cerious Systems will support two launch paths that share the same workspace, market-data services, study engine, sim/live exchange abstraction, alerts, orders, fills, and layout persistence.

1. Browser Portal
The browser portal is the first entry point. Today it runs locally and later it will be served from the cloud portal. The launch page shows Cerious branding, accepts a username and password, and starts the authenticated terminal session. The portal does not start market subscriptions until the user has authenticated.

2. Windows Thin Client
The Windows thin client is a small launcher that opens the same authenticated terminal in a desktop-style app window. In the current local build it starts the local Cerious services, opens the terminal with the desktop-client flag, and uses the same saved workspace data. Later this launcher will authenticate to the cloud session and connect to services running near the exchange data center.

3. Workspace Migration
The Ted S workspace recovered from the 09:15 stable workspace set is the desktop seed layout. On first desktop-client launch, all widgets are minimized so the trader can place them deliberately. After the trader saves, the layout is stored locally for fast load and backed up server-side for recovery.

4. Layout Persistence
The terminal saves the active layout locally and posts a server-side workspace snapshot through the gateway. Local state gives fast startup. Server-side snapshots protect the layout from browser storage loss and allow future cloud/session roaming.

5. Market Data and Studies
CME currently uses Databento through the price-services layer. REST backfill establishes baseline chart/study state, while live market data updates last traded price, top of book, DOM, and rolling OHLCV state. The UI consumes normalized service output and does not introduce product-specific pricing rules.

6. Production Direction
The thin client remains deliberately thin. It should authenticate, launch the terminal, restore workspace state, and connect to service APIs. Market-data plugins, product definitions, study calculations, order state, fills, alerts, and PnL remain service-layer responsibilities so future exchanges such as Kalshi, Polymarket, and Coinbase can be added without rewriting the workspace UI.

Immediate Test Workflow
1. Launch the local browser portal.
2. Log in with the configured Cerious local credential.
3. Confirm the Ted S workspace is available and that current data refreshes through the existing services.
4. Download or run the Win64 thin client package.
5. Confirm first desktop launch opens minimized windows.
6. Arrange windows and save workspace.
7. Relaunch and confirm the saved layout loads locally and is backed up server-side.
